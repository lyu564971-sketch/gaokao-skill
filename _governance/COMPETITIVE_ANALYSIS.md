## 一、对方做对了什么（精华，值得借鉴）

### 1. 槽位采集器（Slot Collector）—— 最值得抄的设计
对方不一次性让用户填表，而是**多轮追问采集结构化槽位**：
- 必采槽位：省份、分数/位次、选科、家庭背景、就业诉求
- 采齐才给建议，不全就继续追问
- "敢说真话" = 在追问中刺破幻想（如"你家做金融吗？没有就别报金融前端"）

**→ 我们要吸收**：把当前 SKILL.md 的"输出规范"前置一个**槽位采集阶段**。用户输入不全时，Agentic Protocol 不直接出报告，而是先反问补齐槽位。这正是我们 CHECKPOINT 第③问"家庭条件用上了吗"的工程化升级。

### 2. 17 模块知识库 + 792 专业速查
对方内置 792 个专业的速查卡（就业方向/薪资/坑点），作为 LLM 的**事实底座**，降低幻觉。

**→ 我们已有**：`references/occupational-data.md`（主流专业速查底座）。但覆盖度不够。**升级方向**：扩充到覆盖高考主流 50-80 个专业类，每个带"中位数起薪/对口单位/避坑点"三件套。

### 3. 模型无关（model-agnostic）
对方支持 DeepSeek/通义/Kimi 等多家，只换参数不换代码。

**→ 我们已做到**：Provider 抽象层（LLMProvider 接口 + OpenAI 兼容）。这点我们更优——我们还抽象了 DataProvider。

### 4. 搜索兜底（百度/Tavily）
对方知识库覆盖不到的，调搜索 API 兜底，并在结果标注 `[联网]`。

**→ 我们已有**：Agentic Protocol Step2 强制 WebSearch + 来源分级（🟢🟡🟠⚪）。我们更严格——对方是"兜底"，我们是"强制先查"。

### 5. 自然语言解析
"一万三"→13000 位次，"不学计算机"→ 排除约束。对方用 LLM 做意图解析。

**→ 我们要吸收**：在槽位采集阶段加一层**自然语言归一化**，把口语转成结构化字段。

## 二、对方的不足（我们要规避）

### 1. 单机运行（Python + 本地 HTML），无 Web 部署
对方要装 Python、双击 bat、本地开服务器。**门槛高，无法分享给他人用**。

**→ 我们的优势**：Next.js + Vercel 部署，浏览器打开即用，送子域名，可分享。这是核心差异化。

### 2. 无流式输出
对方是请求-响应模式，长报告要干等。

**→ 我们的优势**：SSE 流式 + 分阶段进度（分类中→查数据中→生成报告中）。专业感更强。

### 3. 数据黑盒，无可信度分级
对方"42万条录取数据"无来源标注，用户无法验证。

**→ 我们的优势**：每条数据强制五字段（含来源 URL + 可信度等级），UI 显示徽章。这正是你强调的"严格可靠专业"。

### 4. 无 Provider 抽象，搜索逻辑硬编码
对方搜索调用直接写在 agent.py 里，换搜索引擎要改代码。

**→ 我们的优势**：DataProvider 接口 + registry，换引擎只改环境变量。

### 5. 角色定义偏"人设"而非"认知框架"
对方的 system_prompt 偏性格描写（"老炮""情绪是活的"），缺少结构化心智模型。

**→ 我们的优势**：nuwa 12 模块标准，三大铁律 + 局限 + 内在矛盾。更可复现、更工程化。

## 三、升级行动清单（落实到本项目）

| # | 借鉴点 | 落地动作 | 优先级 | 归属阶段 |
|---|--------|---------|--------|---------|
| U1 | 槽位采集器 | SKILL.md 新增"槽位采集阶段"，Agentic Protocol 在分类前先判断槽位是否齐全，不全则反问 | 高 | P1（改 SKILL.md）|
| U2 | 自然语言归一化 | 槽位采集时把"一万三"→13000、"不学计算机"→排除约束 | 中 | P2 |

## 四、对比总览

| 维度 | xuefeng-agent | 本项目（升级后）|
|------|--------------|----------------|
| 部署形态 | 单机 Python + 本地 HTML | **Web 部署，浏览器即用** |

| 数据规模 | 42 万条录取 | 暂无库（联网为主，可选建库）|

## 五、结论

对方验证了"槽位采集 + 知识库 + 搜索兜底"的产品形态可行，但它的工程化（单机/无流式/无分级/无抽象）有明显短板。**我们的架构在工程化上全面领先，只需吸收它的"槽位采集器"设计精华**（U1），即可在产品力上超越。

不盲目照搬它的 42 万数据库——获取官方 Excel 成本高且涉及数据时效维护，我们先用"强制联网 + 冷启动底座 + 可信度分级"的轻量方案，等 MVP 跑通再评估是否建库（U4）。

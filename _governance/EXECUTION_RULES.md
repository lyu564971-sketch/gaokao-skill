# 执行规则（EXECUTION_RULES.md）

> 项目的工作流约定。所有阶段、所有协作方（含 AI agent）必须遵守。

---

## 1. 任务进度日志规范

**文件**：`_governance/PROGRESS_LOG.md`

**规则**：
- 每完成一个阶段任务（或一个有意义的子任务）→ 在 `PROGRESS_LOG.md` 顶部追加一条
- **最新在最上**（倒序），最新进度永远在文件最顶部，便于一眼看到
- 单条格式：

```
## YYYY-MM-DD HH:MM | [阶段编号] 阶段名
- **状态**：🟢完成 / 🟡进行中 / 🔴阻塞
- **产出文件**：
  - path/to/file1.md
  - path/to/file2.ts
- **关键决策/发现**：一句话
- **下一步**：明确下一动作
```

- 未完成的不写"完成"；阻塞必须写明阻塞原因和需要的动作

---

## 2. Bug 审查闭环规范

**文件**：`_governance/BUG_AUDIT.md`

**触发条件**：任何报错、异常行为、与预期不符的输出、用户反馈的问题。

**规则**：
- 发现 bug → 立即在 `BUG_AUDIT.md` 追加一条（编号自增 `BUG-XXX`）
- 必须写全 **5 个字段**，缺一不可：
  1. **现象**：观察到什么（用户/系统视角）
  2. **触发条件**：如何复现
  3. **根因**：为什么会发生（不是"哪里错了"，是"为什么会错"）
  4. **修复方案**：具体改了什么文件/代码
  5. **防再发措施**：具体可执行的工程手段（禁止"以后注意"这类空话）
     - ✅ 合格：加 JSON Schema 校验 / 加 timeout 兜底 / 加单元测试覆盖该分支
     - ❌ 不合格：以后小心 / 多检查 / 注意一下
- 修复完成 → 标注 ✅ 闭环，写明验证方式
- 未修复 → 标注 🔴 待处理，写明阻塞点

## 4. 质量门控（每个阶段交付前自检）

每个阶段进入"完成"状态前，必须逐条勾选：

- [ ] 产出文件已创建且内容完整
- [ ] `PROGRESS_LOG.md` 已追加一条
- [ ] 如遇 bug，`BUG_AUDIT.md` 已立条并闭环
- [ ] 代码无 `any` 滥用（TS 严格模式下必须有明确类型）
- [ ] 错误路径有兜底（不能裸抛、不能静默吞）
- [ ] 关键决策已记入 `DECISIONS.md`（如有新决策）

---

## 5. Provider/接口可插拔约定（架构铁律）

- 业务代码**只依赖** `lib/providers/types.ts` 的接口，**禁止直接 import** 具体实现
- 切换 provider 只改 `.env` 的 `*_PROVIDER` 变量，不改业务代码
- 新增 provider 必须实现完整接口，并在 `registry.ts` 注册
- 这条规则保证"方便后续接入新 API"的可维护性

## 6. 文档与代码同源

- `ARCHITECTURE.md` 是架构的唯一真相源；代码与它不一致时，**以代码为准并更新文档**（不允许文档滞后说谎）
- `SKILL.md` 是角色定义的唯一真相源；前后端共用，禁止硬编码角色逻辑到代码里
- 任何架构偏离必须走 `DECISIONS.md` 立条说明原因

## 7. 环境约束（Windows + 中文环境）

- **项目路径必须为纯 ASCII**（字母/数字/连字符），禁止中文/空格/特殊字符。Windows 下 npm/Node/Git 对非 ASCII 路径的编码处理不可靠（见 BUG-001/BUG-004）。
- 当前项目路径：`D:\gaokao-skill`
- **环境前置检查**：每次开工先跑 `node -v && npm -v` 自检，npm 损坏则阻塞（见 BUG-002 修复方案）。
- **TS 语法约束**：在引入 Next.js（自带 SWC 转译）之前，`.ts` 文件避免使用参数属性（`constructor(private x)`）、enum、namespace 等需编译器转换的语法。P1 后此约束自动解除（见 BUG-003）。

## 7.5 工作流纪律（防卡死/防反复重启）

> 本节由自我审查得出（见会话中断复盘），硬约束。

1. **"继续"指令 = 直接做下一个动作**。不复述进度、不复述计划、不重复 todo 已有内容。最多一句话点明"正在做 X"。
2. **调研型 agent 必须用 `run_in_background`**，或自己用 WebSearch 分步查；禁止把 600s 级同步任务和轻活混排在一条消息里。
3. **工具调用遵循"独立才并行"**：有依赖关系必须串行；单条消息内并行工具 ≤3 个，且都得秒级返回。
4. **Edit 前文件若刚被改过，只 Read 一次目标段落**（带 offset/limit），不整文件重读。
5. **TodoWrite 在阶段切换时立即同步**，不等 reminder 催。
6. **每回合先做、后说**：把工具调用和执行放前面，长篇解释放后面或省略。

- 真实 Key 禁止写入 `.env.example`（占位值：`your-api-key-here`）
- 真实 Key 禁止提交到 Git（`.gitignore` 含 `.env` 规则）
- 完整 Key 值仅在 `_governance/PROGRESS_LOG.md` P8 条目和 Render Dashboard 中可查

### 9.3 部署操作权限

- Render 服务创建：通过 Dashboard Blueprint 链接或 API（API 需绑卡验证）
- Render 环境变量：通过 Render API `PUT /v1/services/{id}/env-vars/{key}` 管理
- Render 部署触发：通过 Render API `POST /v1/services/{id}/deploys` 或 Git push（autoDeploy）
- 服务删除/重建：需用户确认（不可逆操作）

## 10. 远程构建失败排查纪律（P9 教训沉淀）

> 来源：2026-06-20 P9，Render 构建连续 6 次失败，**4 次错误假设被证伪 + 1 次浏览器自动化尝试失败**，浪费大量部署配额。以下规则用于防止重犯。

### 10.1 第一动作永远是「获取真实日志」，不是「猜测推送」
- 远程构建（Render/Vercel/CI）失败时，**第一件事是拿到报错原文**，不是改代码重新推送
- "本地能过"**不能**作为排除代码问题的依据——本地环境 ≠ 线上环境（Node 版本、OS、内存、缓存策略都可能不同）
- 连续猜测推送前，问自己：**"我有报错原文吗？"** 没有 → 先去拿日志，禁止盲推

### 10.2 耗时数据是 API 唯一可靠信号，优先据此缩范围
- Render API 不暴露构建日志（`/logs` 端点 404），但 `startedAt`/`finishedAt` 给了耗时
- **install 阶段挂（~20-30s）** vs **build 阶段挂（~100s+）** 是完全不同的根因方向：
  - install 挂 → 怀疑 lockfile/依赖解析/Node 版本/网络
  - build 挂 → 怀疑代码/编译器/资源
- 排查时**先按耗时归类**，再针对性假设，不要混为一谈

### 10.3 假设证伪后必须换方向，禁止同维度重试
- 一个假设被证伪（如"Node 版本不对"22 失败），**禁止在同维度换值重试**（如再试 24）
  - 本次教训：22 失败后仍试 24，浪费 1 次部署，结果必然同方向证伪
- 证伪后应**切换维度**（Node→依赖→配置→环境变量→缓存），或回到 §10.1 拿真实日志
- 设定**假设预算**：连续 2 次同维度失败，强制停止盲推，转去取日志

### 10.4 浏览器自动化复用登录态——障碍清单与止损阈值
- 自动化读取需登录的页面（如 Dashboard）时，复用真实登录态的成本很高：
  - Chrome `--remote-debugging-port` 拒绝默认 user-data-dir（安全限制）
  - Cookie 在 `Network/Cookies` 子目录（非根目录），且**加密存储**，`strings` 抓不到
  - 很多站点登录态在 localStorage/sessionStorage，无法用文件复制迁移
  - Playwright 自带浏览器与系统浏览器版本可能不匹配，需额外下载
- **止损阈值**：遇到 ≥2 个连续障碍，立即转交用户手动操作（复制文本/截图），不要在工具化上死磕
- 让用户**复制文本**优于**截图**（OCR 丢字符，尤其错误码/路径/标点）

### 10.5 验证完毕才声明，部署配额是有限资源
- 免费版 Render 部署有配额限制，每次盲推都在消耗配额
- 推送前必须本地复现失败（§4 质量门控），本地能复现才能定向修复
- 本地无法复现的远程失败 → **禁止继续推送**，转去取真实日志（§10.1）
